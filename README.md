# SPBPU-low-level-prog

Привет, студент!

Это репозиторий, в котором ты будешь делать лабы по низкоуровневому программированию.

По мере прохождения курса я буду обновлять этот репозиторий. Тебе нужно будет периодически подтягивать изменения.

Желаю удачи!

Добровольно-принудительно мы используем подход, когда мы не загружаем файлы в репозиторий через GUI, а используем git на компьютере. Для того, чтобы начать работу, нужно:
1) Сгенерировать и добавить ssh-ключ (команда ssh-keygen -t ed25519 -C "your_email@example.com");
2) Найти этот ключ на компьютере, скопировать и добавить через GUI github/gitlab.

Добавить ключ в github можно, зайдя в настройки и открыв пункт "SSH and GPG keys"
Добавить ключ в gitlab можно, зайдя в настройки и открыв пункт "Access" -> "SSH keys"

После добавления ключа ты сможешь как устанавливать себе локально репозитории, так и отправлять в них изменения.
Чтобы установить репозиторий локально, нужно сделать следующее:
1) Зайти на страницу репозитория;
2) Нажать на кнопку "Code";
3) Выбрать SSH и скопировать ссылку;
4) В консоли перейти в нужную тебе директорию и выполнить команду:
  git clone *ссылка*

После этого создастся директория, в которой будет этот репозиторий.

Немного терминологии и советов по работе с Git:

- у тебя есть твой удаленный репозиторий, который хранится на github. Ты можешь найти его в своих проектах. То, что ты делаешь локально, не отображается на сайте до тех пор, пока:
а) ты не запушишь в новой ветке свои изменения (тогда на сайте можно выбрать свою ветку и увидеть изменения);
б) ты не выполнишь pull/merge request для того, чтобы подсоединить изменения из твоей ветки в ветку main;
в) я не подтвержу эти изменения.

После этой процедуры твои изменения окажутся в ветке main твоего удаленного репозитория.

- ветки (branches) - это механизм изоляции изменений. Когда ты создаешь одну ветку, ты перестаешь влиять на другие ветки. В рамках этой ветки можно делать абсолютно разные изменения, которые никак не скажутся на другие ветки. 
Хотя в рамках нашего курса тебе вряд ли придется делать много веток, тебе нужно как минимум создать одну, поскольку в ветку main нельзя пушить напрямую. Создать ветку можно так:
git switch -c *branch_name* 
или
git checkout -b *branch_name*

Твоя ветка будет содержать все те файлы (изменения), которые есть в той ветке, на которой ты стоял перед созданием новой.

Чтобы подтягивать изменения из удаленного репозитория, нужно сначала подтянуть инфу об этих изменениях, а затем уже выполнить слияние. Делается это так:
git fetch origin
git pull --rebase 

флаг --rebase в данном случае означает, что сначала подтянутся все коммиты из origin, а затем сверху положатся твои коммиты. Если ты не делал никаких коммитов, можно просто сделать pull.

Когда я делаю изменения в своем удаленном репозитории, тебе нужно сделать следующие операции, чтобы подтянуть эти изменения себе локально:
1) В GUI github нажать на кнопку Update Fork - подтянуть изменения из главного (моего) удаленного репозитория в свой удаленный репозиторий;
2) сделать git fetch origin
3) сделать git pull

Чтобы отправить лабу на реквест, нужно сделать следующее:
1) создать новую ветку (git switch или git checkout);
2) изменить/добавить файлы;
3) добавить изменения/файлы в индексирование git (команда git add *filename*);
4) сделать коммит (git commit -m "message");
5) сделать пуш (git push origin *branch_name*).

Некоторые полезные команды:

git status - показывает, на какой ветке ты находишься, в каком состоянии она находится, какие файлы изменены и т.д.
git log -N - покажет N последних коммитов
git restore *filename* - позволяет заресторить файл, если ты еще не закоммитил изменения

Если ты вдруг сделал коммит и затем понял, что что-то надо поправить - ничего страшного. Ты можешь пойти следующими путями:
1) Простой путь: ты можешь сделать корректные изменения и затем сделать git commit --amend. Тогда ты добавишь эти изменения к тому же коммиту, который создал в прошлый раз. Я рекомендую использовать этот способ и лишний раз не отправлять на ревью сразу несколько коммитов;
2) Сложный путь: можно воспользоваться git reset. Чтобы не утерять изменения, я рекомендую использовать soft версию:
git reset --soft HEAD~1 - эта команда вернет тебя в предкоммитное состояние, когда файлы уже были добавлены с помощью git add, но еще не были вкоммичены. Собственно, сам коммит убирается.
3) Если ты все же сделал два коммита, скорее всего, git предложит тебе сделать git squash. Это позволяет слить два коммита в один. Он выполняется в интерактивном формате, но придется поразбираться что происходит.

Исходя из этого, чтобы упростить себе жизнь, я рекомендую придерживаться правила не делать лишних коммитов и исправлять их по первому описанному выше случаю.
